# Publication Policy For Project Roasts

## Default stance

Project roasts are useful, but they can expose strategy, weak points, founder intent, or private context.

The default rule is:

```text
methodology can be public;
named detailed roasts require consent or strict public-source discipline.
```

## Public-safe formats

Safe to publish:

- methodology
- anonymized patterns
- scorecard templates
- public-source-only review
- founder-approved case study
- redacted examples without private context

Risky to publish without permission:

- private conversation details
- direct critique of a named friend or partner
- unpublished strategy
- screenshots from private chats
- non-public business assumptions
- sensitive product weaknesses

## Consent checklist

Before publishing a named-project roast, confirm:

- project owner knows it will be public
- project owner has seen or approved the public version
- private conversation details were removed
- public links are accurate
- tone is sharp but not humiliating
- the output includes constructive next steps

## Recommended labels

Use one of these labels in each case:

- `public_source_review`
- `founder_approved_case`
- `private_working_note`
- `anonymized_case_pattern`
- `consent_required_draft`

## ISR note

The ISR roast can be used as the first internal prototype of the method.

Until explicit public approval exists, treat it as:

```yaml
publication_status: consent_required_draft
source_mode: public_material_plus_private_context
public_version_allowed: methodology_only_or_redacted_case
```
